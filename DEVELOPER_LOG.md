# سجل تطوير PWA فريق ألفا

هذا الملف يوثّق قرارات وبنية مشروع `pwa/` — التطبيق الذي حلّ محل `team-apk`
(React Native). الهدف: لو رجعت لهذا المشروع بعد فترة، أو احتجت تكمل معه
بمحادثة جديدة، هذا الملف يلخّص "وين وصلنا ولماذا" بدون إعادة شرح كل قرار.

---

## ⚠️ تصحيح لاحق مهم: term أصبح id خام (لا رابط كامل)

بعد جولات التطوير الأولى، تقرر تبسيط `term` بالكامل: **`getGiscusTerm`
يرجع الآن فقط الـ`id` الخام** (مثل `e60f33ee-8538-4962-afa4-57aa8f6fdd47`)،
وليس رابط `/share/...` الكامل كما كان سابقاً.

**السبب:** `data-term` في giscus هو نص حر للمطابقة فقط (giscus يبحث عن
Discussion بعنوان يحتوي هذا النص) — لا يُفتح ولا يُتحقق منه كرابط فعلي.
فاستخدام رابط كامل لم يكن ضرورياً من الأساس، وكان هو سبب كل مشكلة تطابق
web/PWA السابقة (اختلاف `/app` بالمسار). الـid وحده ثابت تماماً بغض النظر
عن الدومين أو base path أو إعادة تسمية الموقع مستقبلاً. هذا التصحيح طبَّقه
المستخدم بنفسه بعد سؤال مباشر، وكان صحيحاً 100%.

**يوجد الآن دالتان منفصلتان بكل من `web` و`pwa`:**

```ts
getGiscusTerm('project', id)  // → فقط الـid الخام، لـterm في <GiscusComments>
getShareUrl('project', id)    // → رابط /share/... كامل، لـCopyLinkButton وOGMetaTags فقط
```

**خطأ كان موجوداً بالـPWA وأُصلح أثناء هذا التصحيح:** كانت `ProjectDetailPage`
و`PostDetailPage` تستخدمان `getGiscusTerm` لكل من `term` *و* `shareUrl`
(نفس القيمة لغرضين مختلفين) — هذا غلط حتى بالتصميم القديم.

**نقطة دقيقة في `getShareUrl` بالـPWA:** يستخدم `SHARED_CONTENT_ORIGIN`
(بدون `/app`)، **وليس** `PRODUCTION_ORIGIN` (بـ`/app`) — لأن الـCloudflare
Function الحقيقية المسؤولة عن معاينة OG (`functions/share/projects/[id].js`)
موجودة فقط بمشروع `web`، لا يوجد مجلد `functions/` مكرر تحت `/app/`
بالـPWA. لو استُخدم `PRODUCTION_ORIGIN` هنا، الرابط الناتج لن يجد أي
Function تستجيب له.

**ملاحظة:** مناقشة GitHub التجريبية الوحيدة الموجودة (مشروع لم يُنشر للعموم
بعد) انفصلت عن مشروعها بعد هذا التعديل — مقبول، تم تأكيده مع المستخدم.

---

## لماذا هذا المشروع موجود

كان عندنا 3 واجهات: `web` (موقع)، `admin-apk` (لوحة إدارة)، `team-apk`
(تطبيق React Native للزوار). تقرر إلغاء `team-apk` بالكامل لصالح **PWA**
مستقل بنفس التصميم والأفكار، لأسباب:

1. **giscus (نظام التعليقات) أسهل بكثير بالويب** — React Native لا يدعم
   iframe، فكان لازم نظام معقد (`embed-comments.html` + WebView) لعرض
   التعليقات. بالـPWA، نفس مكوّن `@giscus/react` يعمل مباشرة بدون أي وسيط.
2. **offline + يعمل على الكمبيوتر** — مطلوب صريح من صاحب المشروع، ومتاح
   بسهولة عبر `vite-plugin-pwa` (Service Worker + Workbox) بدون كتابة كود
   offline من الصفر.
3. **صيانة أبسط** — مشروع ويب واحد (Vite/React) بدل مشروعين بتقنيتين
   مختلفتين (React Native + React Web).

---

## القرار المعماري الأهم: نشر تحت `/app/`

- الموقع الرئيسي (`web`) يبقى على نفس الدومين بمساره الجذري `/`.
- الـPWA يُنشر تحت **نفس الدومين**، بمسار فرعي `/app/`.
- مثال: `https://alpha-team.alghaybmhmd606.workers.dev/app/`

**لماذا رابط واحد بدل دومينين؟** قرار صاحب المشروع صريح: "رابط واحد،
والتعقيدات الفنية تترك لي". الحل: `base: '/app/'` في `vite.config.ts`
(يضبط كل المسارات الداخلية تلقائياً)، و `scope`/`start_url` بنفس القيمة
في PWA manifest حتى لا يتعارض الـService Worker مع أي شيء مستقبلي على
الموقع الرئيسي.

**خطوة النشر المتبقية (لم تُنفَّذ بعد):** نسخ `pwa/dist/` إلى داخل مجلد
`app/` بمشروع `web` (أو الريبو المنشور) قبل رفعه — تماماً كما ننسخ
`functions/` لـ`dist/` في مشروع `web` (نفس فكرة "خطوة دمج وقت النشر").

## ⚠️ خطأ حرج اكتُشف وأُصلح: term الـgiscus

أول نسخة من `GiscusComments.tsx` استخدمت `VITE_SITE_URL` (الذي يحمل
`/app` في آخره) لبناء `term` الخاص بـgiscus. هذا كان سيُنشئ **GitHub
Discussion منفصلة** لكل مشروع/منشور حسب كونه فُتح من `web` أو من الـPWA —
يفقد كل فائدة توحيد التعليقات.

**الحل:** متغير بيئة مستقل `VITE_SHARED_CONTENT_ORIGIN` (بدون `/app`)
يُستخدم **فقط** داخل `GiscusComments.tsx` (term + رابط ثيم CSS)، يجب أن
يطابق حرفياً `VITE_SITE_URL` في `web/.env`. **لو غيّرت دومين الموقع
مستقبلاً، حدّث هذا المتغير في كلا المشروعين معاً.**

```
# pwa/.env
VITE_SITE_URL=https://<domain>/app              # للروابط الداخلية بالـPWA
VITE_SHARED_CONTENT_ORIGIN=https://<domain>      # بدون /app — لgiscus فقط
```

## البنية والتقنيات

| الجانب | الاختيار | السبب |
|---|---|---|
| Build tool | Vite + React + TS | يطابق مشروع `web`، خبرة مألوفة |
| Styling | Tailwind v4 (`@tailwindcss/vite`) | لا حاجة لـ`tailwind.config.js`، إعداد أبسط |
| PWA | `vite-plugin-pwa` (Workbox) | معياري، offline جاهز بأقل كود |
| Routing | `react-router-dom` (`basename="/app"`) | يطابق نمط `web` |
| Data | نفس Supabase الموجود (REST مباشر، لا SDK) | مصدر بيانات واحد بين كل الواجهات |
| State (ثيم/مفضّلة) | React Context (وليس `useState` محلي كما في team-apk) | تزامن فوري بين Sidebar/BottomTabs والصفحات |
| HTML المنشورات | `dangerouslySetInnerHTML` + `DOMPurify` | تحسين عن team-apk (الذي يستخدم `stripHtml` ويفقد التنسيق) — أمان محفوظ بالتنقية |

## التخطيط المتجاوب (الفرق الأساسي عن team-apk)

- **هاتف (`< md`):** `BottomTabs` ثابت بالأسفل — نفس شكل team-apk الأصلي.
- **كمبيوتر (`>= md`):** `Sidebar` جانبي ثابت — قرار صريح من صاحب المشروع
  (شريط سفلي يبدو ضيقاً وسط شاشة كبيرة).
- مصدر واحد لعناصر التنقل: `src/lib/navigation.ts` — تعديل/إضافة تبويب
  بمكان واحد ينعكس على كلا المكوّنين.

## ما تم نقله من team-apk (كامل)

كل الشاشات (19 شاشة أصلية) + كل الـhooks + كل المكوّنات المشتركة:

**الصفحات:** Home, Projects, ProjectDetail, Blog, PostDetail, Downloads,
Versions, More, About, Roadmap, Independent, Glossary, FAQ, Join,
Favorites, Settings, Search, NotFound.

**أهم التحسينات عن النسخة الأصلية (وليس نقل حرفي):**
- `GiscusComments` يعمل مباشرة (لا embed-comments/WebView).
- محتوى المنشورات (`PostDetailPage`) يُعرض كـHTML منسّق فعلي بدل نص خام.
- `useSettings`/`useFavorites` Context حقيقي بدل `useState` محلي.
- التمرير اللامحدود بالمدونة عبر `IntersectionObserver` (معادل ويب لـ
  `onEndReached`) بدل أي حل آخر.
- المشاركة عبر `navigator.share()` (Web Share API) بدل `expo-web-browser`.

## ما لم يُنقل / لم يُبنَ بعد

- **لا إشعارات Push فعلية.** مفتاح الإشعارات بصفحة الإعدادات موجود كـ
  UI/تفضيل محلي فقط (يُخزَّن بـlocalStorage)، لكن لا توجد آلية إرسال Push
  حقيقية مربوطة بعد (تحتاج Push API + إذن متصفح + سيرفر إشعارات).
- **لا اختبار بصري حقيقي بمتصفح فعلي** — كل التحقق تم عبر `tsc -b` (صفر
  أخطاء) و `vite build` الفعلي الناجح (صفر تحذيرات) + فحص مخرجات `dist/`
  والـmanifest يدوياً. لم يُفتح بمتصفح حقيقي لرؤية التصميم بصرياً.
- **لم يُنشر فعلياً تحت `/app/`** — يحتاج نسخ `dist/` لمجلد `app/` بمشروع
  `web` المنشور + التأكد من إعداد `VITE_SHARED_CONTENT_ORIGIN` بالقيمة
  الصحيحة وقت البناء.

## خطوات النشر (لم تُنفَّذ، للمرجع المستقبلي)

```bash
cd pwa
npm install
npm run build          # يولّد dist/ مع sw.js + manifest.webmanifest

# دمج وقت النشر — مثل آلية functions/ في مشروع web:
cp -r dist/* ../web/app/   # أو أي مسار مناظر بالريبو المنشور فعلياً
```

تأكد قبل البناء أن `.env` يحتوي:
- `VITE_SUPABASE_URL` / `VITE_SUPABASE_ANON_KEY` — نفس قيم `web/.env`
- `VITE_SITE_URL` — ينتهي بـ`/app`
- `VITE_SHARED_CONTENT_ORIGIN` — **بدون** `/app`، مطابق لـ`VITE_SITE_URL`
  في `web/.env` حرفياً

## مراجعة شاملة ثانية — فجوات اكتُشفت بعد فحص منهجي

بعد البناء الأول، طرح المستخدم سؤالاً مباشراً: "هل لا يوجد شيء ناقص؟". هذا
دفع لمقارنة منهجية كاملة بين كل صفحة في `web` ونظيرتها في `pwa` (بدل
الاعتماد على ذاكرة الجلسة السابقة)، وكشف عدة فجوات حقيقية:

### تصحيح مهم: لا توجد مشكلة في كون giscus term رابطاً

الملخص السابق وصف استخدام رابط كـ`term` في giscus بأنه "خطأ حرج". هذا غير
دقيق: `data-term` في giscus هو **نص حر للمطابقة فقط** (يبحث عن مناقشة
بعنوان يحتوي هذا النص) — وليس رابطاً يُفتح أو يُتحقق منه. المشكلة الفعلية
الوحيدة كانت كون القيمة **مختلفة** بين `web` والـPWA (بسبب `/app` في
المسار)، وهذا قابل للحدوث بأي تصميم term طالما القيمتان غير متطابقتين
حرفياً — لا علاقة جوهرية بـ"رابط مقابل id بسيط".

### صفحات/ميزات كانت غائبة تماماً، أُضيفت الآن:

- **`/contact`** (ContactPage) — لم تكن موجودة في team-apk أصلاً.
- **`/featured`** (FeaturedProjectsPage) — كذلك.
- **قسم "تعريبات مستقلة" في HomePage** — لم يكن منقولاً.
- **CopyLinkButton + useCopyLink** — لم يُنقَلا في الجولة الأولى، أُضيفا
  لـProjectDetailPage وPostDetailPage. **مهم:** يستخدمان
  `VITE_SHARED_CONTENT_ORIGIN` (بدون `/app`) عبر `getGiscusTerm`، تماماً
  كـ`web` — وليس `VITE_SITE_URL`.
- **"منشورات ذات صلة" + روابط الوسوم لصفحة البحث** في PostDetailPage.

### صفحات أُعيد بناؤها لتطابق نسخة `web` الأحدث (لا تطابق team-apk القديم):

- **GlossaryPage**: جدول + فلتر حسب المشروع (كان بطاقات بدون فلتر).
- **IndependentPage**: أضيف زر "اقترح تعريباً" وحقل المُعرب.
- **JoinPage**: محتوى مختلف بالكامل (6 أدوار بدل 4، نص مختلف) — هذا تطوير
  حصل في `team-apk` بعد `web`، وليس العكس؛ القرار: نطابق نسخة `team-apk`/
  الأحدث (المستخدمة هنا)، و`web` سيُحدَّث بجلسة لاحقة ليطابقها.
- **RoadmapPage**: إحصائيات سريعة + ترتيب حسب أولوية الحالة (لا
  `display_order`) + وصف المشروع بكل بطاقة.
- **DownloadsPage**: أُعيد بناؤها بالكامل — جدول لكل التحميلات الفردية (لا
  تجميع "آخر إصدار لكل مشروع" فقط)، + بحث + فلتر مشروع + إحصائيات + تحميل
  تدريجي (PAGE_SIZE=20، أُضيف `useDownloadsPage` جديد لهذا).
- **ProjectsPage**: أُضيف تبديل عرض شبكة/قائمة (grid/list) — كان غائباً.
  حافظنا على حقل البحث (تحسين زائد عن `web`، وليس نقصاً).
- **SearchPage**: أُعيدت كتابتها بالكامل — اكتُشف أن `web` **لا يستخدم**
  RPC الـ`search_content` رغم وجوده بقاعدة البيانات (كود يتيم/orphaned في
  `web/src/hooks/useSearch.ts`، غير مستورد من أي صفحة). الفعلي هو فلترة
  محلية على مشاريع+منشورات+مستقلة معاً. حُذف `useSearch.ts` من الـPWA
  واستُبدل بنفس منطق `web`.
- **PostDetailPage**: تصحيح جوهري — `post.content` هو **نص شبه-Markdown**
  مفسَّر سطراً بسطر (`## `, `- `)، **وليس HTML خام**. النسخة الأولى
  استخدمت خطأً `DOMPurify` + `dangerouslySetInnerHTML` (كان سيعرض `##`
  حرفياً بدل عنوان منسّق). أُزيلت `dompurify` من المشروع كلياً.
- **AboutPage**: أُضيفت إحصائيات (مشروع/نشط/مكتمل/عضو). أُهملت عمداً
  مكوّنات `Timeline` و`GearDecoration` (الأخيرة زخرفة SVG بحتة بلا وظيفة)
  لتوفير وقت الجلسة — قابلة للإضافة لاحقاً لو احتاجها أحد.

### خطأ تقني اكتُشف وأُصلح وقت التعديل (لا علاقة بـweb):

حذف `useDownloads.ts` (fetch-all) أثناء استبدال `DownloadsPage` بالنسخة
المرقّمة (`useDownloadsPage`) كسر `ProjectDetailPage` و`VersionsPage` بدون
قصد — كلتاهما تحتاجان **كل** تحميلات مشروع واحد بالذات (سياق مختلف عن
صفحة `/downloads` المرقّمة). تم إعادة إنشاء `useDownloads.ts`؛ الآن
يتعايش الهوكان معاً لسياقين مختلفين فعلاً.

### أداة جديدة أُضيفت احتياطياً: `lib/assetUrl.ts`

منقولة من `web` — تحل مسارات الأصول المطلقة (`/images/x.png`) بشكل صحيح
تحت أي `base` (هنا دائماً `/app/`). لا استخدام حالي يتطلبها بشكل حرج (لا
توجد مسارات صور مطلقة مكتوبة يدوياً بالكود حتى الآن)، أُضيفت توقياً لمشاكل
مستقبلية، واستُخدمت فعلياً في عرض القائمة الجديد بـProjectsPage
(`onError` fallback).

### ما لم يُفحص بعد (نطاق متبقٍ محتمل لجلسة لاحقة)

المقارنة المنهجية أعلاه غطّت كل الصفحات المشتركة الأسماء بين `web` والـPWA.
لم تُفحص بعمق: مكوّنات `web` الأصغر (`Timeline`, `GearDecoration`,
`OGMetaTags` بكل صفحة بدقة كاملة)، ولا `AdminPage`/`AdminManageTab` (غير
ذي صلة بالـPWA كموقع زوار). يُنصح بنفس أسلوب المقارنة المنهجية (لا الاعتماد
على الذاكرة) لو استُكمل العمل مستقبلاً.

## ملاحظات للمتابعة

- لو أُضيف حقل جديد بقاعدة بيانات Supabase (مثلاً عمود جديد بجدول
  `projects`)، حدّث `src/types/index.ts` هنا **وبـ`web` معاً** — لا تزامن
  تلقائي بين النوعين حالياً.
- أي تعديل على هوية الألوان (Bronze/Gold) يجب أن يحدث بثلاث ملفات معاً:
  `pwa/src/index.css`، `web/src/index.css` (أو ما يعادله)، وملفات
  `public/giscus/*.css` — لا مصدر ألوان واحد مشترك فعلياً بين المشروعين
  بعد (تحسين مستقبلي محتمل: نقل الألوان لحزمة/ملف مشترك).
